SPEED IMPROVEMENTS OVER MATLAB
--------------------------------
  1. PARALLEL CHAINS (main speedup)
     Use --parallel flag.  All chains run simultaneously on separate CPU cores
     using Python multiprocessing.  With 10 chains on 10 cores → ~10x speedup.
     MATLAB ran all chains sequentially (single process).

  2. NUMPY VECTORISED OPERATIONS
     All array operations use NumPy (C-compiled), avoiding MATLAB overhead.

  3. FAST FILE I/O
     Chains saved as .npz (NumPy compressed binary).  ~10x faster than
     MATLAB's .mat format.  Config saved as JSON (human readable).

  4. NO TOOLBOX DEPENDENCY
     MATLAB required the Statistics Toolbox (prctile, histcounts, randsample).
     Python uses only numpy/scipy/matplotlib (free and open-source).

ALGORITHM (unchanged from MATLAB)
-----------------------------------
  Transdimensional Markov Chain Monte Carlo (RJMCMC) with Parallel Tempering.
  Model: Voronoi partition of the 1D earth into layers.
  Unknowns: number of layers, layer depths, layer resistivities, noise scaling.
  Prior on resistivity: log-uniform [0.1, 1e5] ohm-m.
  Prior on depth: log-uniform [10 m, 350 km].
  Likelihood: Gaussian (Z-impedance) or log-Gaussian (app-res/phase).
  Proposals: birth, death, move, change-rho, change-noise.
  PT: swap models between hot and cold chains to escape local minima.

DATA FORMATS SUPPORTED
-----------------------
  MT impedance   : MT_data_Z.dat    (period, Re(Z), Im(Z), err_Z)
  MT app_phase   : MT_data.dat      (period, app_res, err, phase, err)
  DC Schlumberger: <name>.dat       (AB/2, app_res, err)

HOW TO RUN
----------
  1. Create synthetic data:
       cd MT_Python_code
       python create_synthetic_data.py --noise 3 --nperiods 20 --output data

  2. Run inversion (fast test):
       python run_inversion.py --data data/MT_data_Z.dat \
              --nsteps 20 --nsamples 200 --nchains 2

  3. Run real data (parallel, 10 chains):
       python run_inversion.py --data MT_data_Z.dat \
              --nsteps 1000 --nsamples 1000 --nchains 10 --parallel

  4. Process and plot results:
       python postprocess/chain_convergence.py --folder results
       python postprocess/process_chains.py    --folder results
       python postprocess/plot_posterior.py    --folder results
       python postprocess/plot_noise.py        --folder results

  5. Install dependencies:
       pip install -r requirements.txt

ESTIMATED TIME (Python vs MATLAB)
----------------------------------
  MATLAB (sequential, 10 chains, 1000 steps × 1000 samples):  ~12 hours
  Python  (parallel,  10 chains, 1000 steps × 1000 samples):  ~1-2 hours
  (estimate based on NumPy speed + 10-core parallelism)

  Generated a third synthetic MT dataset for inversion testing, with a more

MODEL (geologically motivated: Proterozoic mobile belt / rifted margin)

  Layer 1:      0 –   2 km,    20 ohm-m   (sediment / weathered cover)
  Layer 2:      2 –  15 km,  2000 ohm-m   (resistive upper crust)
  Layer 3:     15 –  40 km,    50 ohm-m   (lower-crust conductor)
  Layer 4:     40 – 150 km,  3000 ohm-m   (resistive lithospheric mantle)
  Layer 5:    150+ km,         30 ohm-m   (asthenospheric conductor)

ACQUISITION PARAMETERS
----------------------
  Periods : 20 log-spaced, 0.1 – 10 000 s  (covers sediment → asthenosphere)
  Noise   : 4 % Gaussian on Z
  Seed    : 42 (default in create_synthetic_data.py)

OUTPUT
------
  data/synthetic/mobile_belt/
      MT_data_Z.dat            (period, Re(Z), Im(Z), err_Z)
      MT_data.dat              (period, app_res, err, phase, err)
      true_model.json          (layer tops + resistivities)
      synthetic_data_plot.png  (app-res and phase, true vs noisy)

COMMAND USED
------------
  python create_synthetic_data.py \
         --noise 4 --nperiods 20 --pmin 0.1 --pmax 10000 \
         --layer_tops 0 2000 15000 40000 150000 \
         --layer_rho  20 2000 50 3000 30 \
         --output data/synthetic/mobile_belt

WHY THIS TEST IS USEFUL
-----------------------
  * Alternating conductor-resistor sequence → checks that RJMCMC can resolve
    multiple interfaces instead of collapsing to a smoother 3-layer model.
  * Broad period range (0.1 – 1e4 s) probes depths from ~1 km to > 200 km,
    covering crust, lithospheric mantle, and asthenosphere — the same
    depth span targeted in the Southern African study.
  * 4 % noise is moderately high, giving a realistic challenge for
    noise-hyperparameter estimation (change_noise / change_noise2 moves).

[2026-04-15]  BUG FIX: cell_birth sort order + forward NaN handling
--------------------------------------------------------------------------------

SYMPTOM
-------
  On the mobile_belt 5-layer synthetic test (10 steps × 1000 samples × 5 chains)
  the inversion became very slow compared with the default 3-layer case.
  Chains 1 and 2 spent ~13 000 forward-model calls per step to accept 500
  samples (acceptance rate ~3.5 %), and the log was filled with:

    mt_forward.py:57: RuntimeWarning: overflow encountered in exp
    mt_forward.py:60: RuntimeWarning: invalid value encountered in scalar multiply
    mt_forward.py:61: RuntimeWarning: invalid value encountered in scalar divide

ROOT CAUSE
----------
  cell_birth() inserted the new Voronoi node *after* the nearest existing
  node. The "nearest" neighbour is chosen by absolute distance, so it may
  be deeper than the proposed depth z_new. When that happened, the model's
  internal depth column became UNSORTED, and model_to_layers() then produced
  NEGATIVE thicknesses via np.diff(z).  Negative h in mt1d_forward turned
  exp(-2*h*dj) into exp(+large) → overflow → Inf → NaN in the impedance.
  The NaN propagated into compute_residual(), giving a NaN likelihood.
  Python's builtin min(0.0, NaN) returns 0.0 (unlike MATLAB's min), which
  caused Metropolis-Hastings to accept some NaN proposals and then reject
  nearly everything afterwards, wasting forward-model calls.

FIX
---
  1. mcmc/proposals.py :: cell_birth
         The new node is now inserted at its correct sorted position using
         np.searchsorted(internal_depths, z_new), not at nearest+1.
         This guarantees np.diff(z) ≥ 0.

  2. forward/mt_forward.py :: mt1d_forward
         Wrapped the period loop in
             np.errstate(over='ignore', under='ignore',
                         invalid='ignore', divide='ignore')
         to silence the (now much rarer) spurious warnings and avoid the
         Python warnings-machinery overhead per forward call.

  3. mcmc/likelihood.py :: compute_residual
         If LF is non-finite (NaN or Inf), return LF = 1e30 and nrms filled
         with a large finite sentinel.  This ensures any residual pathological
         proposal is cleanly rejected by MH instead of corrupting the chain.

  4. run_inversion.py :: _write_acceptance_summary
         New argument elapsed_sec.  The Acceptance_Rate_Summary.txt now
         includes a "Total time" line (h/m/s and minutes).

EXPECTED IMPACT
---------------
  * No more "overflow / invalid value" warnings during inversion.
  * Per-step acceptance rates become consistent across chains (previously
    chain 1/2 were stuck at 3–4 % because of repeated NaN rejections).
  * Forward-model call count per step should drop by ~3–4× on hard cases.
  * The summary file now records wall-clock time, which we will quote in
    the thesis benchmark section.

       kPDF = nansum(S.kSamples,2)./(nansum(nansum(S.kSamples))*S.dz);

AUTOMATION DRIVER  (new: Bayesian_code/run_all_processing.m)
-------------------------------------------------------------
  Loops over every <STATION>_R1/ folder in Bayesian_code/ and runs
  A_01 → A_02 → A_03 inside each, with these properties:

     * cd's into the station folder before each script (chain files are
       loaded relative to cwd).
     * Calls each script BY NAME via a small helper function that does
       `eval(scriptname)`, NOT via MATLAB's run() — because run() silently
       cd's into the script's own folder while executing.  That behaviour
       was the cause of an A_02 "MT_DC_TD_Chain_001.mat not found" error
       even though the chain files were present and A_01 (loaded via the
       same run()) had appeared to succeed.
     * Helper-function isolation: A_02 starts with `clc; clear; close all`,
       which would wipe the driver's loop variables if executed in the
       same workspace.  Calling each script from inside a helper function
       confines the `clear` to the helper's local workspace.
     * Adds the station name as figure super-title (sgtitle) before
       saving, so each image is self-labeled.
     * Original per-station outputs (MT_TD_Chain.jpg, MT_Post_image.jpg)
       remain inside <STATION>_R1/.
     * Driver additionally writes station-prefixed quick-look copies into
       a top-level Bayesian_code/output/ folder
       (<station>_convergence.jpg, <station>_posterior.jpg) for
       side-by-side review of all sites.

DEBUGGING DISCOVERY (worth recording for thesis methods chapter)
-----------------------------------------------------------------
  MATLAB's `run(fullpath)` function temporarily changes the working
  directory to the script's containing folder while the script executes,
  then restores cwd on return.  Scripts that load data files using bare
  filenames ("relative-to-cwd" loads, e.g. load('MT_TD_Chain_001.mat'))
  therefore look in the SCRIPT's folder rather than the caller's working
  directory — the opposite of what is usually expected.  Two safe ways
  to invoke a script from a driver:
     (a) addpath the script's folder once, then call by name
         (e.g. eval('A_01_see_chain_convergence')) — cwd is preserved.
     (b) use load(fullfile(data_dir, 'file.mat')) inside the script —
         independent of cwd.
  This explains why every script in Process_All_Chains worked when typed
  manually at the prompt (the user's pwd was the station folder) but
  failed under the driver until we switched away from run().
